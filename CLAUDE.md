# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> **Repo context:** This is **JarvisPanda** — a personal autonomous AI stack built on top of [OpenJarvis](https://github.com/open-jarvis/OpenJarvis), a Stanford-originated open-source fork of the OpenJarvis framework. The `src/openjarvis/` package *is* OpenJarvis. The `deploy/` directory is the JarvisPanda-specific Discord bot, worker, observer, and Docker stack layered on top of it.

## Architecture

OpenJarvis is a **local-first personal AI framework** organized around five core primitives. The Python package lives in `src/openjarvis/`, with a Rust workspace in `rust/` that exposes functionality to Python via PyO3 bindings (`rust/crates/openjarvis-python`).

### Five Primitives

1. **Intelligence** (`src/openjarvis/intelligence/`) — Model catalog and metadata. Maintains `BUILTIN_MODELS` with specs (VRAM, context length, etc.). Runtime-discovered models merge into `ModelRegistry`.

2. **Engine** (`src/openjarvis/engine/`) — Inference backends. All implement `InferenceEngine` ABC (`generate()`, `stream()`, `list_models()`, `health()`). Supported: Ollama, vLLM, SGLang, llama.cpp, MLX, Cloud (OpenAI/Anthropic/Google). Discovery probes all backends and falls back automatically.

3. **Agents** (`src/openjarvis/agents/`) — Pluggable agent implementations. Hierarchy: `BaseAgent` → `ToolUsingAgent` → specific agents. Available: `SimpleAgent`, `OrchestratorAgent`, `NativeReActAgent`, `NativeOpenHandsAgent`, `RLMAgent`, `OpenHandsAgent`, `ClaudeCodeAgent`, `OperativeAgent`, `MonitorOperativeAgent`. The `SandboxedAgent` wrapper (in `sandbox/`) runs any agent inside Docker/Podman.

4. **Memory** (`src/openjarvis/memory/`) — Persistent, searchable storage. Backends: SQLite/FTS5 (default), FAISS, ColBERTv2, BM25, Hybrid (RRF fusion). When `agent.context_from_memory` is enabled, relevant docs are retrieved and prepended to prompts.

5. **Learning** (`src/openjarvis/learning/`) — Trace-driven feedback loop. Router policies (`heuristic`, `learned`, `sft`, `grpo`) learn from accumulated traces which model/agent/tool combos perform best per query type.

### Registry Pattern

All extensible components use a decorator-based registry defined in `src/openjarvis/core/registry.py`. To add a new backend, implement the appropriate ABC and decorate with the registry:

```python
from openjarvis.core.registry import EngineRegistry

@EngineRegistry.register("my_engine")
class MyEngine(InferenceEngine):
    ...
```

Registries: `ModelRegistry`, `EngineRegistry`, `MemoryRegistry`, `AgentRegistry`, `ToolRegistry`, `RouterPolicyRegistry`, `BenchmarkRegistry`, `ChannelRegistry`.

### EventBus

All primitives communicate through a thread-safe pub/sub `EventBus` (`src/openjarvis/core/events.py`). Subscribers are called synchronously in registration order. Key event types: `INFERENCE_START/END`, `TOOL_CALL_START/END`, `MEMORY_STORE/RETRIEVE`, `AGENT_TURN_START/END`, `TRACE_STEP/COMPLETE`, `SECURITY_SCAN/ALERT/BLOCK`.

### Configuration

Config is TOML-based. File location: `~/.openjarvis/config.toml` (generated by `jarvis init`). JarvisPanda's active runtime config lives at `deploy/docker/config.persistent.toml`.

Sections map to primitives. Key fields:

```toml
[engine]
default = "ollama"           # ollama | vllm | llamacpp | sglang | cloud

[engine.ollama]
host = "http://localhost:11434"

[intelligence]
default_model = "qwen3:8b"
fallback_model = "llama3.2:3b"
temperature = 0.7             # generation params live here, NOT under [agent]
max_tokens = 1024

[agent]
default_agent = "orchestrator"
max_turns = 10
tools = "web_search,think,memory_store,memory_search"
context_from_memory = true    # auto-inject relevant memory into every query

[tools.storage]
default_backend = "sqlite"    # sqlite | faiss | colbert | bm25 | hybrid
context_top_k = 5
context_min_score = 0.1
context_max_tokens = 2048

[learning]
enabled = false
[learning.routing]
policy = "heuristic"          # heuristic | learned | sft | grpo

[security]
enabled = true
mode = "warn"                 # warn | redact | block

[server]
host = "0.0.0.0"
port = 8000
agent = "orchestrator"
```

**Engine selection priority:** `--engine` CLI flag → `intelligence.preferred_engine` → `engine.default` → first healthy engine discovered.

**Migration notes:** `temperature`/`max_tokens` moved from `[agent]` → `[intelligence]`. `[memory]` renamed to `[tools.storage]`. Learning policies moved from flat fields to nested `[learning.routing]`/`[learning.intelligence]`/`[learning.agent]` sub-sections. Old formats still load via backward-compatible properties.

Full reference: `docs/getting-started/configuration.md`.

### Key Supporting Systems

- **Traces** (`src/openjarvis/traces/`) — SQLite-backed recording of full agent interactions (steps, routing, inference calls, tool use).
- **Telemetry** (`src/openjarvis/telemetry/`) — Per-call inference metrics (latency, energy, GPU stats), stored in SQLite and published on EventBus.
- **Server** (`src/openjarvis/server/`) — OpenAI-compatible FastAPI HTTP API (`/v1/chat/completions`, `/v1/models`, `/health`).
- **MCP** (`src/openjarvis/mcp/`) — Model Context Protocol layer.
- **Scheduler** (`src/openjarvis/scheduler/`) — Cron/interval/once task scheduling with SQLite persistence.
- **Channels** (`src/openjarvis/channels/`) — Messaging platform integrations (WhatsApp via Baileys Node.js bridge, plus Telegram, Discord, Slack, etc. via optional deps).
- **Security** (`src/openjarvis/security/`) — Secret/PII scanning, `GuardrailsEngine` wrapper, audit logging.
- **Evals** (`src/openjarvis/evals/`) — Benchmarking framework with TOML-configured eval runs.

### Agent Selection Guide

All agents implement `BaseAgent.run()` returning an `AgentResult`. Choose based on task needs:

| Agent key | Class | When to use |
|---|---|---|
| `simple` | `SimpleAgent` | Single-turn Q&A, no tool use |
| `orchestrator` | `OrchestratorAgent` | General multi-step tool use; function-calling loop |
| `native_react` | `NativeReActAgent` | When reasoning traces are required; Thought-Action-Observation pattern |
| `native_openhands` | `NativeOpenHandsAgent` | Code generation/execution inline (CodeAct); also supports tool `Action:` fields |
| `rlm` | `RLMAgent` | Document-scale contexts; stores long context as REPL variables |
| `openhands` | `OpenHandsAgent` | Full OpenHands SDK wrapper for AI-driven software development (Python 3.12+) |
| `claude_code` | `ClaudeCodeAgent` | Spawns `@anthropic-ai/claude-code` Node.js subprocess; requires Node.js 22+ + `ANTHROPIC_API_KEY`. In JarvisPanda, Tier 3 work is done via GitHub Actions + Claude Code GitHub App instead. |
| `sandboxed` | `SandboxedAgent` | Wraps any agent in Docker/Podman with disabled networking + mount allowlists |

`OrchestratorAgent` is the standard choice for the JarvisPanda stack. `ClaudeCodeAgent` is available for direct use but in JarvisPanda Tier 3 escalation goes through the Claude Code GitHub App (subscription OAuth) rather than this class directly.

The `SandboxedAgent`'s `ContainerRunner` blocks sensitive path patterns (`.ssh`, `.env`, `*.pem`, cloud configs). Use it for untrusted or user-submitted code execution.

### Memory Backends

All backends implement `MemoryBackend` (`store()`, `retrieve()`, `delete()`, `clear()`). Switch via `[tools.storage] default_backend`:

| Backend | Config key | Deps | Quality | Persistence | Notes |
|---|---|---|---|---|---|
| SQLite/FTS5 | `sqlite` | none | good | ✓ disk | Default; BM25 ranking |
| FAISS | `faiss` | `faiss-cpu`, `sentence-transformers` | high | ✗ memory | Dense cosine similarity; `all-MiniLM-L6-v2` (384-dim) |
| ColBERTv2 | `colbert` | `colbert-ai`, `torch` | highest | ✗ memory | Late interaction MaxSim scoring; heavy deps, slow |
| BM25 | `bm25` | `rank_bm25` | good | ✗ memory | Lightweight; rebuilt each operation |
| Hybrid | `hybrid` | depends on sub-backends | highest | depends | RRF fusion; retrieves `top_k * 3` then re-ranks |

**Chunking** (`ChunkConfig`): 512 token chunks, 64 overlap, 50 min. Splits on paragraph boundaries (`\n\n`).

**Context injection** (`ContextConfig`): top_k=5, min_score=0.1, max_context_tokens=2048. Retrieved results prepended as system messages with source attribution tags.

Per-user memory in JarvisPanda: every Discord query is prefixed `[Discord User: name | ID: N]` and the agent is instructed to tag `memory_store` calls with `user:{ID}:` in the source/metadata field. Query with `_search_user_memory(query, user_id)` in `deploy/discord/bot.py`.

### Deep Research Pattern

The `OrchestratorAgent` + `memory_store`/`memory_search` multi-turn loop is the standard for research tasks:

```python
from openjarvis import Jarvis

j = Jarvis(model="qwen3:8b", engine_key="ollama")
response = j.ask(
    "Research this topic in depth:\n\n{topic}",
    agent="orchestrator",
    tools=["web_search", "arxiv_search", "think", "memory_store", "memory_search"],
    max_turns=15,
    temperature=0.5,
)
```

The agent iterates: search → store findings → search more → recall stored → synthesise. The `think` tool allows reasoning without side effects. `memory_store`/`memory_search` provide cross-turn scratch space. Loop terminates at `max_turns` or when the agent decides it has enough.

The built-in `deep-research` skill (`src/openjarvis/skills/data/deep-research.toml`) codifies this pattern as a reusable TOML pipeline. Use `!run-skill deep-research <topic>` in Discord.

### Scheduler Tasks

`TaskScheduler` (`src/openjarvis/scheduler/`) polls SQLite for due tasks and calls `system.ask()` on each. Three schedule types:

```python
from openjarvis.scheduler import SchedulerStore, TaskScheduler

store = SchedulerStore("/data/scheduler.db")
scheduler = TaskScheduler(store, system=jarvis_instance, poll_interval=60)
scheduler.start()

# Cron task
scheduler.create_task(
    prompt="Summarise today's top AI news and store to memory",
    schedule_type="cron",
    schedule_value="0 9 * * *",   # 9am UTC daily
    agent="orchestrator",
    tools="web_search,think,memory_store",
)

# Interval task (seconds)
scheduler.create_task(
    prompt="Check for new arXiv LLM papers",
    schedule_type="interval",
    schedule_value="3600",        # every hour
)

# One-shot
scheduler.create_task(
    prompt="...",
    schedule_type="once",
    schedule_value="2026-04-01T09:00:00+00:00",
)
```

In JarvisPanda, the scheduler is embedded in `jarvis-discord`. Two tasks are seeded at startup (daily morning brief, weekly AI digest). Task results are posted to `DISCORD_DIGEST_WEBHOOK` via an `EventBus` `scheduler_task_end` subscription. Users manage tasks via `!tasks`, `!schedule`, `!pause-task`, `!resume-task`, `!cancel-task`, `!task-log` Discord commands. The scheduler DB lives at `/data/scheduler.db` in the shared volume.

Enable via the proper config section (maps to `JarvisSystem.scheduler`):
```toml
[scheduler]
enabled       = true
db_path       = "/data/scheduler.db"
poll_interval = 60
default_agent = "orchestrator"
```

**Pause vs cancel:** `pause_task()` halts execution temporarily — `resume_task()` recomputes `next_run` from now and re-activates. `cancel_task()` is permanent (sets `next_run = null`, status = `cancelled`).

**Operator metadata:** pass `system_prompt` + `operator_id` in `metadata` for per-task behaviour customisation:
```python
scheduler.create_task(
    prompt="...", schedule_type="cron", schedule_value="0 9 * * *",
    metadata={"system_prompt": "Be concise.", "operator_id": "daily-brief"},
)
```

Cron parsing uses `croniter` if installed (added to Dockerfiles), with a minute-granularity fallback. Full CLI also available: `jarvis scheduler create/list/start/pause/resume/cancel/logs`.

### Messaging Hub / Channels

OpenJarvis has 26+ built-in channel integrations registered in `ChannelRegistry`. Each implements `BaseChannel` with `ChannelMessage` / `ChannelHandler` / `ChannelStatus`. Add via `jarvis add <platform>` or environment variables:

```bash
jarvis channel list    # show available channels
jarvis channel status  # show configured channels
```

**Slack example:**
```env
SLACK_BOT_TOKEN=xoxb-...
SLACK_APP_TOKEN=xapp-...
```

Channel TOML recipe pattern:
```toml
[channel]
default = "slack"

[agent]
type = "orchestrator"
max_turns = 5
temperature = 0.3
tools = ["think", "memory_store", "memory_search"]
```

JarvisPanda currently uses a custom `discord.py` bot (`deploy/discord/bot.py`) rather than the built-in `discord_channel` integration — this gives finer control over thread context, per-user memory prefixing, and the `CommandRegistry` pattern. Migrating to `ChannelRegistry` is a future option that would enable Telegram/Slack/WhatsApp in a single config change.

## JarvisPanda Stack

This repo is both the OpenJarvis SDK (`src/openjarvis/`) and a live deployment of it. The deployment runs as five Docker services defined in `deploy/docker/docker-compose.persistent.yml`. The active runtime config is `deploy/docker/config.persistent.toml`. Credentials go in `deploy/.env` (copy from `deploy/.env.example`).

### Docker services

| Service | Role |
|---|---|
| `jarvis-server` | OpenAI-compatible HTTP API on `:8000` |
| `jarvis-discord` | Discord bot — runs the full OrchestratorAgent in-process |
| `jarvis-digest` | Hacker News AI digest, posts to Discord webhook every N days |
| `jarvis-claude-worker` | Label manager — ensures `claude-code-*` labels exist, optional webhook receiver. AI work is done by GitHub Actions. |
| `jarvis-observer` | Health watchdog — reads trace/telemetry DBs, auto-opens fix issues, posts daily arXiv digest |

```bash
cd deploy/docker
docker compose -f docker-compose.persistent.yml up -d --build
docker compose -f docker-compose.persistent.yml logs -f
```

Shared SQLite volumes at `/data/`: `memory.db`, `telemetry.db`, `traces.db`, `scheduler.db`.

### Three-tier escalation

```
Tier 1  qwen3:8b via Ollama       — handles all queries locally
   ↓  stuck after 4 turns on a technical task
Tier 2  Gemini Flash              — second opinion / implementation plan
   ↓  Gemini confirms code changes needed
Tier 3  GitHub issue (claude-code-work label) → GitHub Actions → Claude Code GitHub App
         → implements fix on branch → opens PR → Discord notified
```

Never call `github_issue` without first trying `gemini_escalate`. See `AGENTS.md` for the full agent guide (tool reference, memory rules, what not to do).

### PR lifecycle

```
github_issue created (claude-code-work label)
  → GitHub Actions fires (.github/workflows/claude-issue-worker.yml)
  → @claude comment posted on issue (Claude Code GitHub App, subscription OAuth)
  → Claude Code implements fix on branch, opens PR
  → Discord notified via DISCORD_DIGEST_WEBHOOK
  → local agents verify and merge: !merge <pr#>  or  👍 reaction
```

The `jarvis-claude-worker` container only ensures labels exist — it no longer runs AI itself. All AI implementation work happens in GitHub Actions via the Claude Code GitHub App (no `ANTHROPIC_API_KEY` needed; uses Claude Max subscription).

### Key environment variables

| Variable | Required | Description |
|---|---|---|
| `DISCORD_BOT_TOKEN` | ✓ | Discord bot token |
| `GEMINI_API_KEY` | ✓ (Tier 2) | Gemini Flash escalation |
| `GITHUB_TOKEN` | ✓ (worker/observer) | PAT with repo scope |
| `GITHUB_REPO` | ✓ (worker/observer) | `owner/repo` |
| `DISCORD_DIGEST_WEBHOOK` | ✓ | Webhook for digest/observer/Actions posts; also set as GitHub Actions secret |
| `TAVILY_API_KEY` | | Richer web search (falls back to DuckDuckGo) |
| `GITHUB_WEBHOOK_SECRET` | | Optional: instant issue pickup via webhook (expose `GITHUB_WEBHOOK_PORT=9000`) |

**Note:** `ANTHROPIC_API_KEY` is NOT required. Tier 3 AI work uses the Claude Code GitHub App via OAuth (Claude Max subscription). Set `DISCORD_DIGEST_WEBHOOK` as both a Docker env var and a GitHub Actions secret (`Settings → Secrets → Actions`).

Full variable list: `deploy/README.md`.

## Conventions

- **Commit messages**: Conventional Commits format (`feat:`, `fix:`, `docs:`, `test:`, `refactor:`), under 72 chars, reference issues with `fixes #N`.
- **Pytest markers**: `live` (requires running engine), `cloud` (requires API keys), `nvidia`/`amd`/`apple` (hardware-specific), `slow`.
- **Node.js**: Required only for `ClaudeCodeAgent` (`src/openjarvis/agents/claude_code_runner/`) and the WhatsApp Baileys channel bridge. Node.js 22+ required.
- For new PRs that add new primitives, engines, or major refactors, open an issue for discussion first.
- **New tool**: copy `src/openjarvis/tools/weather.py`, register with `@ToolRegistry.register("name")`, import in `tools/__init__.py`.
- **New skill**: copy `src/openjarvis/skills/data/topic-research.toml` — steps chain `tool_name` + `arguments_template` (uses `{placeholder}` substitution) + `output_key`.
- **New Discord command**: add `@cmd.exact` / `@cmd.prefix` decorated async function in `deploy/discord/bot.py` — no changes to event loop needed.
- **New channel**: implement `BaseChannel`, register with `@ChannelRegistry.register("name")`, add credentials to `.env`.
